# temperature 影响回答的随机性
1. 核心作用机制
低 Temperature（低温）：会放大高概率词汇的优势，使概率分布更加集中（尖锐）。模型每次都会倾向于选择最可能出现的词，输出结果更加确定和保守24。
高 Temperature（高温）：会使概率分布变得更加平滑，缩小各个候选词之间的概率差距，让低概率词汇也有更大的机会被选中。这会增加输出的随机性和创造性24。
2. 不同取值范围的影响与适用场景
Temperature = 0：完全确定性输出。模型每次都会选择概率最高的词，重复生成几乎完全相同的内容。适合需要精准、固定结果的场景，如法律条文解读、公式推导、代码生成等16。
0 < Temperature < 1：平衡确定性与多样性。输出既贴合主流逻辑，又有一定的灵活性，不会过于死板。适合大多数日常场景，如日常聊天、写邮件、回答常识问题等14。
Temperature ≥ 1：随机性显著提升。模型会更大概率选择低概率的候选词，输出可能更有创意，但也容易出现逻辑混乱、偏离主题甚至“胡说八道”的情况。适合创意写作、头脑风暴、科幻故事生成等需要激发灵感的场景14。
# role的作用
在大模型（LLM）的 API 接口中，消息列表（Messages）通常由 system、user 和 assistant 三种角色组成。其中，system（系统）角色的 message 具有最高优先级，它的核心作用是为模型设定全局背景、行为准则和指令约束。
你可以把它理解为导演在开拍前给演员（大模型）讲戏，或者给员工下达的“工作手册”。它的具体作用体现在以下几个方面：
1. 设定角色与人设
你可以利用 system message 赋予模型一个特定的身份。模型会在这个身份的框架下回答用户的问题。
示例：“你是一个拥有10年经验的资深Python后端工程师，擅长解决并发问题。”
效果：模型后续的回答会采用专业、严谨的语气，并倾向于提供深度的技术解答。
2. 规定输出格式
当你的程序需要解析大模型的返回结果时，可以通过 system message 强制规定输出的格式。
示例：“无论用户问什么，你必须且只能返回标准的 JSON 格式，不要包含任何额外的解释文本。”
效果：确保接口返回的数据结构稳定，方便下游代码直接解析。
3. 设定全局规则与安全边界
用于限制模型的行为，防止它回答不该回答的问题，或者偏离设定的主题。
示例：“你是一个客服助手。你只能回答与本公司产品相关的问题。如果用户询问政治、天气或其他无关话题，请礼貌地拒绝并引导回产品问题。”
效果：充当“护栏”，保证模型输出的安全性和业务相关性。
4. 提供全局上下文（RAG 常见用法）
在检索增强生成（RAG）架构中，通常会把从数据库中检索到的背景知识放在 system message 中。
示例：“请根据以下参考资料回答用户的问题。如果资料中没有相关信息，请直接回答‘我不知道’，不要自己编造。参考资料：[此处插入检索到的文档内容]”
效果：让模型“开卷考试”，大幅减少幻觉（胡说八道）的概率。
💡 System、User、Assistant 的分工对比：
System（系统）：定规矩、立人设、给背景（“你是谁，你要怎么做”）。
User（用户）：提问题、给指令（“用户现在要你做什么”）。
Assistant（助手）：模型之前的历史回答，用于维持多轮对话的上下文连贯性（“你之前是怎么回答的”）。
总结：system message 是控制大模型行为最有效、最底层的工具。在实际开发中，把核心的指令和约束放在 system 里，比放在 user 里效果要好得多。

# 核心控制参数（决定生成内容的质量与长度）
1. max_tokens（最大生成长度）
作用：限制模型单次回复的最大 Token 数量。
用法：大模型是按 Token 计费的。设置这个参数可以有效控制成本，防止模型“长篇大论”。例如，如果只需要简短回答，可以设为 100；如果需要长文，可以设为 2000 左右。
2. top_p（核采样参数）
作用：与 temperature 类似，用于控制生成文本的多样性，但机制不同。它表示模型只从累积概率达到 top_p 的候选词中采样。
用法：比如设为 0.1，模型就只考虑最有可能的那 10% 的词。注意：官方建议 top_p 和 temperature 二选一调整，不要同时修改，以免产生不可控的干扰。
3. stop（停止词）
作用：定义一个字符串或字符串列表，当模型生成到这些词时，会强制停止输出。
用法：例如设置 stop=["\n\n"]，模型在遇到连续两个换行符时就会停止。这能有效防止模型跑题或输出多余的废话。
二、 工程与体验参数（优化交互体验与输出格式）
1. stream（流式输出）
作用：控制是否开启流式响应（类似打字机效果）。
用法：默认通常是 False（等全部生成完再一次性返回）。在开发聊天机器人时，强烈建议设为 True。这能大幅降低用户的等待延迟，提升交互体验。
2. response_format（结构化输出）
作用：强制模型以指定的格式（通常是 JSON）返回内容。
用法：设置为 {"type": "json_object"}。当你需要通过代码直接解析大模型的返回结果（比如提取实体、生成配置）时，这个参数比在提示词里苦苦哀求“请返回JSON”要稳定得多。
三、 进阶与工具参数（构建复杂应用必备）
1. tools / function calling（工具/函数调用）
作用：允许大模型在对话过程中，根据用户的意图自动调用外部函数或 API。
用法：你可以在参数中定义好函数的名称、描述和参数结构。当用户问“今天北京天气怎么样”时，模型不会瞎编，而是会返回一个指令，要求你的程序去调用天气 API，然后把结果喂回给模型。这是构建 AI Agent（智能体）的核心。
2. seed（随机种子）
作用：保证结果的可复现性。
用法：传入一个固定的整数（如 42）。在相同的输入和参数下，模型每次都会给出完全一致的回答。这在开发调试阶段非常有用。
3. presence_penalty / frequency_penalty（话题新颖度与频率惩罚）
作用：控制模型谈论新话题的倾向，以及减少词语的重复。
用法：正值会惩罚已经出现过的词。如果你发现模型在长篇大论时总是重复同一句话，或者总是绕着同一个话题说，可以适当调高这两个参数（通常在 0 到 2 之间）。
总结建议：
在实际开发中，model + messages + temperature + max_tokens 是最基础的四件套；如果做聊天界面，加上 stream；如果做数据提取，加上 response_format；如果要开发高级 Agent，则必须研究 tools。
